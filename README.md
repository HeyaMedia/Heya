# Heya

Self-hosted media server for films, TV, music, and books — written in Go, ships
as a single binary with the Nuxt SPA embedded. Pairs with
[HeyaMedia/metadata-server](https://github.com/HeyaMedia/metadata-server) for
all upstream metadata (TMDB, TVDB, AniDB, MusicBrainz, OpenLibrary), so this
server stays focused on libraries, playback, and the UI.

## What's in the box

| Area               | What you get                                                            |
| ------------------ | ----------------------------------------------------------------------- |
| Libraries          | Local + SMB mounts, fsnotify + periodic re-scan, NFO-aware matching     |
| Metadata           | Multi-language titles & overviews, cast/crew, keywords, certifications  |
| Playback           | HLS transcoding with per-client capability negotiation, ASS subtitles   |
| Trickplay          | Scrubbing thumbnails generated by the scheduler (off by default)        |
| Background work    | River-on-Postgres job queue, live status over WebSocket                 |
| Auth & users       | bcrypt + session tokens, per-user lists / favorites / watch state       |
| UI                 | Nuxt 4 SPA, dark theme, embedded into the Go binary at build time      |
| CLI                | Cobra-based: scan, identify, transcode, queue, migrate, dashboard TUI  |
| Tailscale          | Optional [tsnet](https://tailscale.com/docs/features/tsnet) integration — joins your tailnet directly, no port forwarding ([docs](docs/tailscale.md)) |

The HTTP API is documented at `/api/docs` (Scalar over auto-generated OpenAPI
3.1 via Huma v2). Real-time events stream at `ws://…/api/events`.

## Architecture in one paragraph

A single Go binary serves both the HTTP API (`net/http` stdlib router) and the
prebuilt Nuxt SPA (`//go:embed dist`). Postgres is the only datastore — it
holds media data, user state, and the River background job queue. Metadata is
fetched exclusively through the
[`metadata-server`](https://github.com/HeyaMedia/metadata-server) aggregator,
so Heya itself never talks to TMDB/TVDB/etc. directly. Transcoding shells out
to ffmpeg with a decision matrix that compares the source streams against the
client's reported capabilities. More detail in
[`docs/architecture.md`](docs/architecture.md).

## Running locally

### Prerequisites

- Go 1.26+
- [Bun](https://bun.sh) — the only JS package manager we use (never npm/pnpm/yarn)
- Docker + `docker compose` (Postgres)
- `ffmpeg` and `ffprobe` on `$PATH` (transcoding + media analysis)
- A reachable [`metadata-server`](https://github.com/HeyaMedia/metadata-server)
  instance, or `https://heya.media` if you don't want to self-host that side

### First-time setup

```bash
git clone git@github.com:HeyaMedia/Heya.git
cd Heya
cp .env.example .env        # tweak the values you care about
make db-up                  # start Postgres on :5440
make build                  # builds frontend (bun) + Go binary into ./bin/heya
./bin/heya setup            # guided wizard — writes .env and creates the admin user
./bin/heya serve            # http://localhost:8080
```

Configuration lives in `.env` (see `.env.example` for every supported key).
Anything declared in env is locked in the Settings UI with a tooltip pointing
back to the env var — `HEYA_LIBRARY_<N>_*` even lets you declare libraries
declaratively for Docker / k8s.

Then add a library via the UI (Settings → Libraries) or CLI
(`./bin/heya library add …`) and Heya will scan, match, and start populating
metadata. Day-to-day development runs `make dev`, which uses **mprocs** to
supervise `heya dev-proxy` on `:8080` (the stable front door), Nuxt on
`:3000`, and the backend (`heya serve --dev-backend`, hot-reloaded by air) on
`:3050`. The front door stays up across air rebuilds, so the browser's HMR
socket and in-flight WS connections never see the backend churn. Requires
`brew install mprocs`.

## CLI quick reference

```bash
./bin/heya serve                  # start the HTTP server
make dev                          # mprocs: dev-proxy :8080 (→ Nuxt :3000 + Go :3050) — open :8080
./bin/heya dashboard              # full-screen TUI: queue, scans, watchers
./bin/heya library list           # show libraries
./bin/heya library scan <id>      # trigger a scan
./bin/heya queue list             # inspect River jobs
./bin/heya transcode probe <file> # ffprobe a file via Heya's analyzer
./bin/heya migrate up             # run pending migrations (goose)
./bin/heya --help                 # everything else
```

Full command reference: [`docs/cli.md`](docs/cli.md).
