# Typed API client (OpenAPI → TypeScript)

The Go server's Huma operations generate an OpenAPI 3.1 spec. We commit both
that spec and the generated named TypeScript types so API changes remain
visible in review and ordinary Nuxt builds do not run a code generator.

Transport and server-state management are intentionally separate:

- `web/app/plugins/heyaApi.client.ts` provides the Nuxt-native `$heya` ofetch
  client and owns path expansion, auth/version headers, and 401 logout.
- Pinia Colada query modules under `web/app/queries/` own query keys, caching,
  retry/invalidation, persistence, and sensitivity metadata.
- `web/shared/api/types.gen.ts` supplies response/request types generated from
  the committed API contract.

## Regenerating

```bash
make gen-api-client          # regenerate JSON spec + committed TypeScript types
./bin/heya openapi-spec      # dump just the spec (use -o file to write)
```

The generator is isolated under `web/tools/api-codegen/`. The application uses
TypeScript 7, while the current generator still executes compiler APIs removed
by TypeScript 7; its private TypeScript 5.9 install cannot affect Nuxt.

Lefthook and CI regenerate both artifacts and block drift.

## Files

| File | Role |
| ---- | ---- |
| `web/shared/api.openapi.json` | Full OpenAPI contract generated by Huma. |
| `web/shared/api/types.gen.ts` | Named request, response, and schema types generated from the contract. |
| `web/app/plugins/heyaApi.client.ts` | Injects `$heya`, expands path params, attaches auth/client headers, and handles 401s. |
| `web/app/queries/*.ts` | Pinia Colada query definitions and cache policy. |

## Calling the API

Use `$heya` inside a Colada query or mutation, or for a one-off imperative
action:

```ts
import type { AdminSystemBody } from '~~/shared/api/types.gen'

export const adminSystemQuery = defineQueryOptions(() => ({
  key: ['admin', 'system'],
  query: async () => {
    const { $heya } = useNuxtApp()
    return await $heya('/api/admin/system') as AdminSystemBody
  },
  staleTime: 2_000,
  meta: { persistence: 'none', sensitivity: 'secret' },
}))
```

Path placeholders use the familiar `path` option; query and body are native
ofetch options:

```ts
const detail = await $heya('/api/media/{id}', {
  path: { id: mediaID },
  query: { include: 'credits' },
})
```

The transport is generic because one client serves every Huma route. Import
the generated named type at the query boundary instead of re-declaring a
response shape. Components should normally consume the typed Colada result,
not call the transport directly for server state.

## When to keep raw `$fetch`

Use raw `$fetch` for calls whose response is not ordinary API JSON, including:

- binary streams and image/media bodies;
- multipart uploads;
- WebSocket and SSE connections;
- login/register, where an expected 401 must not trigger `$heya`'s global
  authenticated-session logout behavior.

## Gotchas

- **Spec-vs-app type mismatches**: fix the Huma schema when possible. For a
  deliberate wire-shape exception, cast through `unknown` and document why.
- **`useNuxtApp()` placement**: see the
  [UI conventions](./ui.md#never-call-usenuxtapp-inside-computed-or-async-bodies)
  and hoist it out of computed callbacks.
- **Keep Colada in charge of server state**: do not add a second cache to the
  transport or create ad-hoc component watchers for fetch lifecycle.
